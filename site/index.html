The problem

Distributed systems fail silently

A JWT expires in your Go auth service, cascades through your Java doc service, triggers a retry loop in Node.js — and by the time you investigate, you're staring at 40,000 lines of free-form text across three tabs.

🔍

No shared structure

Go logs with zap, Java logs with SLF4J, Node with Winston — three formats, zero cross-service query capability, and no contract any of them has to honour.

🕳️

Silent message drops

You can't tell the difference between "nothing happened" and "logs were dropped between the service and Kafka." A sequence counter changes that.

🤖

LLM triage is expensive

Sending the same recurring RateLimitExceeded error to your LLM a thousand times a day costs money and time. Fingerprint dedup eliminates ~95% of those calls.

🚨

Errors as strings

A log with error: "connection refused" tells a machine nothing about retryability, error kind, or root cause. Typed Err structs enforce machine-readable failures at the API level.

Architecture

Five components. One contract.

Every service emits the same two-line NDJSON. Fluent Bit compresses and forwards. The brain stores, classifies, and triages — all from typed columns, no full-text search.

Wire format

55-byte header. Zero JSON parsing.

Every language emits the same two-line NDJSON. Part A is a fixed-position header read by the brain at hard-coded byte offsets. Part B is abbreviated JSON — 18 field names compressed to 1–2 characters.

Part A 55-byte fixed-position header — read without JSON parsing
A:1 | trc_7f2a1b9e4d0001 | spn_004 | spn_001 | 04 | 2 | 3 | 2 | 1748268153812
schema_version [0] trace_id [2–15] span_id [17–23] parent_span [25–31] seq [33–34] service [36] level [38] outcome [40] ts_ms [42–54]
Part B Abbreviated JSON — 18 field names shortened to 1–2 chars, ~70% smaller 
{"e":"provider.send.rejected","m":"Provider rejected send","er":{"ek":"RateLimitExceeded","ec":"PROVIDER_QUOTA","em":"quota exceeded","rt":true},"c":{"di":"doc_abc123"},"o":"failure","ms":87,"ui":"usr_xyz","js":"checkout.payment"}
e → event m → message ms → duration_ms er → error ek → error.kind ec → error.code rt → error.retryable c → ctx o → outcome ui → user_id js → journey_stage
AI Triage

From failure to typed fix in milliseconds.

The brain classifies, deduplicates, and routes failures to one of ten typed repair categories — with LLM calls eliminated for 95% of repeated errors.

1

Parse Part A at fixed byte offsets

The Rust brain reads trace_id, level, outcome, and ts_ms from hard-coded positions — no JSON parsing for the 95% of operations that only need typed columns.

2

Gap detection via sequence counter

Per-service atomic seq counter (mod 256) lets the brain detect dropped messages. A 30-second Valkey grace window prevents false positives from out-of-order delivery.

Gap 5 — Valkey 30s TTL
3

Fingerprint dedup — xxHash64 in ~8ns

xxHash64(service_byte | event | error_code | ctx_primary_key). Computed at emit time in all three languages. Cache hit returns the repair immediately — no LLM call.

~95% LLM call elimination
4

Failure classification from typed columns

Isolated · Cascading · ExternalDependency · Timeout · GapDetected · Partial — classified entirely from log_index without touching log_payload.

5

LLM call with constrained output

Call graph + environment snapshot + version-pinned code context assembled and sent to the LLM. Output constrained to one of ten typed RepairIds.

6

Typed RepairId → fix playbook

NETWORK_RETRY · RATE_LIMIT_BACKOFF · VALIDATION_FIX · DATABASE_FIX · AUTH_REFRESH · DEPENDENCY_FALLBACK · CONFIG_FIX · RESOURCE_EXHAUSTED · CONSISTENCY_FIX · UNKNOWN
